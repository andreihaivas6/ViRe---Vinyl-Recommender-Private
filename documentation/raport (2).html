1. Project description

Vire is a system able to "intelligently" recommend by exposing a SPARQL endpoint - vinyl music records according to various criteria: user preferences (specified via controlled natural language constructs such as "I always like/love/prefer classical music, especially opera music by Rossini or Verdi and performed by Angela Gheorghiu or Juan Diego Flórez; I sometimes like progressive rock and post-rock; I like only metal albums released before 2000; I always dislike/hate rap and hip-hop; I dislike songs produced by Flood in the last 25 years"), past song purchases on various music stores, playlists - available online via music streaming services: Last.fm and alternative solutions and locally - for instance, by uploading a JSPF/XSPF document. The playlists could be created by the user or shared by her/his virtual "friends" (consider at least one social network). The system will use several music-related knowledge models and available public resources: Discogs, MusicBrainz, Musicmoz Music Styles.

2. Use Cases

2.1. End Users

The end users of our application are people passionate about music who are looking for vinyl recommandations based on their preferences. Additionally, they want to share their musical preferences with friends or they can make a new playlists together. For example, they might combine their playlists for a party so that everyone can listen to music that they both enjoy.

2.2. Use cases description

Initially, users have the option to either sign up for a new account or log in with an existing one. Once logged in, they have the capability to create music playlists, import existing ones using a JSPF file, or link their last.fm profile.

They can also link their account to their music store profile, which allows them to add songs they've previously purchased. Users also have the opportunity to connect with friends and share their musical tastes, and combine their playlists. The users can also add description for they musical taste for example adding phrases like: "I sometimes like progressive rock and post-rock; I like only metal albums released before 2000; I always dislike/hate rap and hip-hop; I dislike songs produced by Flood in the last 25 years".

The application's main function is to integrate these features to provide personalized vinyl recommendations based on these considerations.

3. Architecture

3.1 Overall Architecture

After analysis the requirementes of the project, we concluded to the following arhitecture with the following components:
  1. The client will interact with our web application. There he will be able to create playlists, share them with friends and also search for any data regarding music. Also, he can link his Music Store and Music Streaming apps accounts to our web application. This way, we can get data from those apps and use it to recommend music to the user.
  2. The web application will communicate with the API Gateway, which will be responsible for routing the requests to the correct microservice. There will be a microservice for each functionality of the web application, in total 3 microservices.
  3. User microservice will be the first microservice and will be responsible for user and friends. It will be responsible for authenticating the user and also for creating new accounts. Also, it will provide user profile and also manages the friends list.
  4. User microservice will communicate with User database. There will be stored all the user data, including the friends list.
  5. Playlist microservice will be the second microservice and will be responsible for playlists. It will be responsible for creating new playlists, adding songs to playlists and also for sharing playlists with friends. Also, it will provide data regarding music, such as: albums, artists, songs.
  6. Playlist microservice will communicate with Playlist database. There will be stored all the playlists data. Also, there will be stored all the data regarding music.
  7. Recommendation microservice will be the third microservice and will be responsible for recommendations. It is responsible for collecting user preferences from its input. Also, it will be responsible for recommending music to the user, based on:
    • user input - regarding preferences;
    • his created playlists;
    • his playlists from Music Streaming apps;
    • his past purchases from Music Store apps.
  8. Recommendation microservice will communicate with Preferences database. There will be stored all the data regarding user preferences. The data will come from:
    • user input - regarding preferences;
    • his created playlists - Daily scheduled analysis over user playlists
    • his playlists from Music Streaming apps;
    • his past purchases from Music Store apps.
  9. Daily scheduled analysis over user playlists will be a scheduled task that will run every day and will analyze the user playlists in order to get more data regarding user preferences.
  10. Cache over Preferences database. It will be used to store the most accessed data from Preferences database. This way, we can reduce the number of requests to the database as we will often need the information regarding users preferences.
  11. Vinyl database will be an internal dataset populated with possible vinyls that can be recommended to the user. The data will be collected from the internet and will be stored in this database.

3.2 Services

In order to get a more understandable view of the architecture, we will dive in every microservice and explain its functionality.

3.2.1 User service

OpenAPI Specification can be found here.

It will be the first microservice and will be responsible for user and friends. It will communicate with User database in order to get the data regarding users and friends. As we can see in the diagram, the functionalities of the microservice are divided into 3 modules:

  1. User module

    This module have the following functionalities:
    • See own profile - the user can access data about his account on our application.
    • See other user profile - users can view the profiles of other members, exploring their details.
    • Search for users - enables users to search and discover other members based on different criteria, such as username
    • See users list - provides a list of users on the platform, allowing exploration and connection with a diverse user base.
    • See friends suggestions - suggests potential connections based on shared interests.

  2. Auth module

    This module have the following functionalities:
    • Authentication - the process by which users confirm their identity, ensuring secure access to the platform and protecting their accounts from unauthorized use.
    • Account creation - enables new users to register and create their accounts on the platform.

  3. Friend module

    This module have the following functionalities:
    • See friends profile - allows users to view detailed information about their friends, including shared playlists within the platform.
    • Add friend - enables users to add a user as a friend/follower
    • Remove friend - provides users with the option to disconnect from friends, removing them from their friends list and limiting shared content.
    • Search friend - allows users to search for specific friends within their network.
    • See friends list - displays a list of the user current friends, facilitating easy access to their profile.

3.2.2 Playlist service

OpenAPI Specification can be found here.

It will be the second microservice and will be responsible for playlists. It will communicate with Playlist database in order to get the data regarding playlists and music. As we can see in the diagram, the functionalities of the microservice are divided into 5 modules:

  1. Playlist module

    This module has the following functionalities:
    • Get playlist info - retrieves information about a specific playlist, including details about its songs.
    • Create playlist - allows users to create new playlists
    • Update playlist - enables users to modify the details and content of an existing playlist
    • Get playlists list - provides a list of playlists available on its profile.
    • Add song to playlist - allows users to add a specific song to a chosen playlist, customizing their music collections.
    • Remove song from playlist - permits users to remove a song from a playlist.
    • Share playlist with friend - let the user to share its content of a playlist with some of its friends
    • Combine playlist - enables users to merge or combine multiple playlists into a single playlist - reunion or intersection
    • Import playlist as JSPF doc - allows users to import playlists in JSPF document format.
    • Delete playlist - allows users to delete a playlist, removing it from their collection and the platform.
    • Search playlist - provides a search functionality for users to find specific playlists.

  2. Song module

    This module has the following functionalities:
    • See song info - allows users to view detailed information about a specific song
    • See songs list - provides a list of songs available on the platform.
    • Search songs by filters - allows users to search for songs based on specific filters

  3. Album module

    This module has the following functionalities:
    • See album info - allows users to view detailed information about a specific album.
    • See albums list - provides a list of albums available on the platform.
    • Search albums by filters - enables users to search for albums based on specific filters.

  4. Artist module

    This module has the following functionalities:
    • See artist info - allows users to view detailed information about a specific artist.
    • See artists list - provides a list of artists available on the platform.
    • Search artists by filters - enables users to search for artists based on specific filters.

  5. Music data collector module

    This module has the following functionalities:
    • Collect music data - collects data regarding music, such as: albums, artists, songs. It has to push this data into our database.

3.2.3 Recommendation service

OpenAPI Specification can be found here.

It will be the third microservice and will be responsible for recommendations. It will communicate with Preferences database in order to get the data regarding user preferences. As we can see in the diagram, the functionalities of the microservice are divided into 6 modules:

  1. App link module

    This module has the following functionalities:
    • Link Music Streaming app - enables users to link their Music Streaming app account to our web application (Last.fm).
    • Link Music Store app - enables users to link their Music Store app account to our web application (Discogs).

  2. Preferences module

    This module has the following functionalities:
    • Collecte preferences - collects user preferences from its text input. There will be involved some processing over the given text with the help of Controlled Natural Language.
      This input will be inserted into our Preferences database.

  3. User data collector from third parties module

    This module has the following functionalities:
    • Collect preferences from music stores and streaming apps - collects user preferences from its playlists from Music Streaming (Last.fm) apps and Music Store apps (Discogs).
      The result will be inserted into our Preferences database.

  4. Daily analysis module

    This module has the following functionalities:
    • Get user preferences from user playlists by daily analysis - collects user preferences from its playlists (from Playlist database) by running some analysis over them. This analysis will be done daily and will be done over the playlists created by the user in the last 24 hours in order to add only new preferences.
      The result will be inserted into our Preferences database.

  5. Recommendation module

    This module has the following functionalities:
    • The recommendation module is divided in two parts:
        Recommendation for input
        The user inserts some preferences and the service will recommend vinyls based on those preferences. For example: I like Queen band. I like rock music. I dislike rap music. We will parse the sentences into tokens and retain the information like this: {'likes': {'artist':{'Queen'}, 'genre':{rock}}, dislikes: {'genre':{rap}}} Then, we will create a query that satisfies all these requests and we will get the vinyls from our Vinyl database and we will recommend them to the user. Also, we used the spotify API to get similar artists with the ones that the user likes to supliment the recommendation module.
        Recommendation for user
        The user will get recommendations based on his preferences such as:
        • his created playlists -
        • XSPF file
        • his playlists from Music Streaming apps;
        • his past purchases from Music Store apps.
        The algorithm works similiar, but with more data about the user history. We will take in consideration the latest user preferences like this: the last day counts the most, then the last week, then the last month and so on.
    • Get recommendations - retrieves recommendations for a specific user, based on his preferences such as:
      • user input - regarding preferences;
      • his created playlists - Daily analysis module
      • his playlists from Music Streaming apps;
      • his past purchases from Music Store apps.
      He will get the preferences from our Preferences database via Cache - as the service will constantly recommend vinyl to the user, we need to reduce the number of requests to the database by storing the preferences in cache.
      Also, he will get the data regarding vinyls to recommend from our Vinyl database.

  6. Vinyl data collector module

    This module has the following functionalities:
    • Collect vinyls data for recommendations - collects data regarding vinyls that can be recommended to the user.
      The data will be collected and will be stored in our Vinyl database.

4.4. RDF

In our project, we utilize RDF (Resource Description Framework) to define a structured vocabulary of terms and relationships between these terms. RDF schema provides a means to precisely specify and organize the terminology and connections used within our project.

We use MusicOntology, a vocabulary for describing music-related information on the Semantic Web. In order to use the Music Ontology, conforming with their guidelines, we need to add a few namespaces to our document 

          
            @prefix mo: http://purl.org/ontology/mo/ .
            @prefix dc: http://purl.org/dc/elements/1.1/ .
            @prefix xsd: http://www.w3.org/2001/XMLSchema# .
            @prefix tl: http://purl.org/NET/c4dm/timeline.owl# .
            @prefix event: http://purl.org/NET/c4dm/event.owl# .
            @prefix foaf: http://xmlns.com/foaf/0.1/ .
            @prefix rdfs: http://www.w3.org/2000/01/rdf-schema#

5. Technologies Used

5.1 Python

Python is a high-level, general-purpose programming language. Its design philosophy emphasizes code readability with the use of significant indentation. Python is dynamically typed and garbage-collected. It supports multiple programming paradigms, including structured (particularly procedural), object-oriented and functional programming. It is often described as a "batteries included" language due to its comprehensive standard library

5.2 Flask

Flask is a micro web framework written in Python. It is classified as a microframework because it does not require particular tools or libraries.[2] It has no database abstraction layer, form validation, or any other components where pre-existing third-party libraries provide common functions. However, Flask supports extensions that can add application features as if they were implemented in Flask itself. Extensions exist for object-relational mappers, form validation, upload handling, various open authentication technologies and several common framework related tools.

5.3 React

React (also known as React.js or ReactJS) is a free and open-source front-end JavaScript library for building user interfaces based on components. It is maintained by Meta (formerly Facebook) and a community of individual developers and companies. React can be used to develop single-page, mobile, or server-rendered applications with frameworks like Next.js. Because React is only concerned with the user interface and rendering components to the DOM, React applications often rely on libraries for routing and other client-side functionality.

5.4 MongoDB

MongoDB is a source-available cross-platform document-oriented database program. Classified as a NoSQL database program, MongoDB uses JSON-like documents with optional schemas. MongoDB is developed by MongoDB Inc. and current versions are licensed under the Server Side Public License (SSPL) which is considered non-free by some organizations and distributions. MongoDB is a member of the MACH Alliance.

5.5 PostgreSQL

PostgreSQL also known as Postgres, is a free and open-source relational database management system (RDBMS) emphasizing extensibility and SQL compliance. It was originally named POSTGRES, referring to its origins as a successor to the Ingres database developed at the University of California, Berkeley. In 1996, the project was renamed to PostgreSQL to reflect its support for SQL.

5.6 Sparql

SPARQL (a recursive acronym for SPARQL Protocol and RDF Query Language) is an RDF query language—that is, a semantic query language for databases—able to retrieve and manipulate data stored in Resource Description Framework (RDF) format. It was made a standard by the RDF Data Access Working Group (DAWG) of the World Wide Web Consortium, and is recognized as one of the key technologies of the semantic web.

5.7 Redis

Redis (Remote Dictionary Server) is an open-source in-memory storage, used as a distributed, in-memory key–value database, cache and message broker, with optional durability. Because it holds all data in memory and because of its design, Redis offers low-latency reads and writes, making it particularly suitable for use cases that require a cache. Redis is the most popular NoSQL database, and one of the most popular databases overall.

5.8 Controlled natural languages

Controlled natural languages (CNLs) are subsets of natural languages that are obtained by restricting the grammar and vocabulary in order to reduce or eliminate ambiguity and complexity. Traditionally, controlled languages fall into two major types: those that improve readability for human readers (e.g. non-native speakers), and those that enable reliable automatic semantic analysis of the language.

6. External Data Sources

To generate recommendations, we require specific data related to users' musical preferences, their self-provided preferences, their existing playlists, and their purchased songs from diverse music stores.

To gather this necessary data, we'll leverage Last.fm, a music streaming service enabling users to curate playlists. Additionally, we'll utilize the Discogs API, a platform providing access to users' past music purchases.

We also use DBpedia, which provides information about various music-related entities, such as artists, albums, genres, and more.

We also use The Music Ontology Specification, that provides main concepts and properties for describing music (i.e. artists, albums and tracks) on the Semantic Web.

7. Linked Data Principles

Linked data is a paradigm on the web that relies on existing web technologies, minimizing redundancy, facilitating reusability, enabling discovery, maximizing interconnectedness between related 'things,' and embracing a pragmatic approach.
Several key principles define linked data:

1. Use URIs as names for things
2. Use HTTP URIs so that people can look up those names.
3. When someone looks up a URI, provide useful information, using the standards (RDF*, SPARQL)
4. Include links to other URIs. so that users can discover more things.

We maintain these principles by assigning unique HTTP URIs to represent vinyl records, artists, genres, and users in the application, by using RDF to define metadata and relationships between these entities, by employing SPARQL for querying user preferences and by establishing links between entities such as vinyls, artists, genres, or songs."
